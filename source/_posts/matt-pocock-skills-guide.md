`mattpocock/skills` 可以把很多原本靠口头提示完成的 Agent 工作流，变成可复用的“操作手册”。它的价值不在于某一句 prompt，而在于把对齐、规划、实现、排障、交接这些工程动作拆成一个个可组合的 skill。

这篇文章只讲最值得先装、先用的一批主力 skill。阅读顺序按真实开发流程安排，适合拿来当团队内的入门说明。

## 一图理解

如果把一轮 AI 协作开发拆开，大致可以分成五段：

1. 先把问题问清楚
2. 再把方案落成文档或 issue
3. 然后按稳定节奏实现
4. 出问题时用固定诊断流程排查
5. 最后把上下文和结果交接出去

Matt Pocock 这套 skill，基本就是围绕这五段来组织的。

## 1. 先解决“Agent 没理解你”

### `grill-me`

这是最适合最先上手的 skill。它会连续追问你的目标、边界、约束和验收标准，直到需求足够清晰。

适合场景：

- 新功能需求还比较模糊
- 你知道想解决什么问题，但说不清实现边界
- 你不想让 Agent 一上来就开始写代码

核心价值：

- 把“模糊想法”变成“可执行说明”
- 提前暴露遗漏条件
- 减少返工

一句话理解：`grill-me` 是需求对齐器。

### `grill-with-docs`

如果项目里已经有 `CONTEXT.md`、ADR、领域文档，优先用这个。它不只是追问你，还会用项目现有术语和决策来挑战你的方案，避免 Agent 写出“看上去合理，但和项目真实语境不一致”的东西。

适合场景：

- 已有复杂代码库
- 团队内有领域语言和设计约束
- 想一边对齐，一边顺手更新文档

一句话理解：`grill-with-docs` 是带项目上下文的高级对齐模式。

## 2. 把想法变成文档和可执行任务

### `to-prd`

当讨论已经足够成熟，但还没形成正式文档时，用它把当前上下文整理成 PRD。

适合场景：

- 聊天里已经有较完整需求
- 想把零散结论沉淀成结构化文档
- 准备给团队或后续 Agent 继续使用

核心价值：

- 降低从讨论到文档的摩擦
- 把验收标准、范围、背景写清楚

### `to-issues`

这个 skill 的目标不是“列任务清单”，而是把方案拆成可以独立认领、独立推进的 vertical slices。

适合场景：

- 已经有 PRD、方案文档或较清晰设计
- 准备进入执行阶段
- 不希望任务拆得只剩技术动作，没有交付价值

核心价值：

- 拆出来的是能交付的切片，而不是流水账
- 更利于并行推进
- 更利于 Agent 按阶段稳定完成

一句话理解：`to-issues` 是把方案切成“可拿走就做”的工程任务。

### `triage`

它用于处理 issue 流转和状态机，不是单纯“看一眼问题”。如果你的团队已经有标签规范、优先级规则、问题分流流程，这个 skill 会很顺手。

适合场景：

- 管理需求池、缺陷池
- 对 issue 做归类、状态推进和责任分发
- 想让 Agent 按统一流程而不是自由发挥

## 3. 真正进入实现阶段

### `tdd`

这是很核心的实现型 skill。它强调 red-green-refactor，而不是“让 Agent 一次性生成一大坨代码”。

适合场景：

- 新功能开发
- 逻辑修改
- 需要高确定性的代码演进

核心价值：

- 控制单次变更范围
- 先证明行为，再扩展实现
- 降低大模型一次性改太多带来的失控风险

一句话理解：`tdd` 让 Agent 按更像资深工程师的节奏写代码。

### `prototype`

当问题还不适合直接进正式实现时，先做原型。这个 skill 很适合用在状态设计、交互验证、信息架构和多个 UI 方向探索上。

适合场景：

- 产品思路还在收敛
- 前端页面需要先看感觉
- 某个模型或状态机需要先跑起来验证

它的重点不是“做得完美”，而是快速验证方向。

### `improve-codebase-architecture`

当项目不是缺一个功能，而是整体开始变得难改、难测、难理解时，用它看架构层面的改进机会。

适合场景：

- 模块耦合严重
- 领域边界模糊
- 想提高可测试性、可导航性、可维护性

这类 skill 更像资深工程师做的“系统性复盘”，而不是普通重构建议。

## 4. 出问题时别靠猜

### `diagnose`

这是排障主力 skill，强调一条很明确的闭环：

`复现 -> 缩小范围 -> 提出假设 -> 打点/观测 -> 修复 -> 回归验证`

适合场景：

- 难复现 bug
- 性能退化
- 看上去像“偶现问题”的复杂故障

核心价值：

- 把排障从“试错”变成“诊断”
- 降低胡乱改代码的概率
- 更容易留下可复用的调查路径

一句话理解：`diagnose` 是给 Agent 上故障处理纪律。

### `zoom-out`

当你或 Agent 已经陷进某个局部文件、局部组件、局部实现细节里，可以用它强制拉高视角，重新看这个模块在整个系统中的位置。

适合场景：

- 对某块代码不熟
- 感觉一直在局部打转
- 需要先理解整体，再决定怎么改

## 5. 把工作交接出去

### `handoff`

这个 skill 适合把当前对话、背景、已完成内容、剩余风险压缩成一份交接材料，让另一个 Agent 或人类开发者接手。

适合场景：

- 长任务需要切换执行者
- 需要保存阶段性上下文
- 不想让下一个人再从头读完整对话

它的意义不只是“摘要”，而是为了降低上下文丢失。

## 6. 还有几类很实用的辅助 skill

### `setup-matt-pocock-skills`

这是安装后的起手式。它会把 issue tracker、标签体系、文档目录这些关键环境先配好。没有这一步，后面的规划和 triage 类 skill 往往会缺上下文。

### `setup-pre-commit`

适合给项目快速补齐提交前检查，比如格式化、类型检查、测试。

### `git-guardrails-claude-code`

适合在 Agent 协作里给危险 git 操作加护栏，降低误推送、误重置、误清理风险。

### `write-a-skill`

当你已经总结出一套适合团队自己的工作流时，可以把它沉淀成自定义 skill，而不是每次重复解释。

### `caveman`

这是沟通风格类 skill，用来压缩 token 消耗，减少冗余表达。适合上下文很长、你只想要短句和结论的场景。

## 推荐安装顺序

如果你第一次接触这套技能，建议按下面顺序体验：

1. `setup-matt-pocock-skills`
2. `grill-me`
3. `to-prd`
4. `to-issues`
5. `tdd`
6. `diagnose`
7. `handoff`

这条路径能最快让你感受到它和普通 prompt 模板的区别：它不是替你“写一段更长的提示词”，而是在给 Agent 一套更稳定的工作方式。

## 我的使用建议

这套 skill 最适合以下团队：

- 已经开始让 Agent 参与真实工程工作
- 不满足于“一次性生成代码”
- 希望把协作流程沉淀成固定套路

如果你只想偶尔让 AI 写个小脚本，未必需要全部装上。但只要你开始在真实项目里让 Agent 参与需求澄清、任务拆解、调试、重构和交接，这套 skill 很快就会变得有价值。

最值得优先建立的不是“更多 skill”，而是这三个习惯：

1. 先对齐，再实现
2. 先拆解，再并行
3. 先诊断，再修改

把这三个习惯带进团队之后，skill 才会真正发挥作用。

## 一个真实的操作例子

为了避免这篇文章只停留在概念层，我在当前博客项目里实际做了一个轻量音乐播放器网页原型，而不是只写说明。目标是回答一个很具体的问题：

这个博客项目里，如果要快速验证一个“浏览器内轻量音乐播放器”的感觉，应该先做成什么样？

我没有直接做成正式功能，而是按 `prototype` skill 的思路，先做一个 throwaway UI prototype。

### 这次我实际做了什么

第一步，我查看了 `prototype` skill 的说明，并选择它的 UI 分支，而不是逻辑分支。因为这次要回答的问题不是“状态机对不对”，而是“这个页面应该长什么样”。

第二步，我没有只做一个页面样式，而是在一个独立路由里做了 3 个差异明显的变体，并通过 `?variant=` 参数切换。这是 `prototype` skill 明确建议的做法：把几个方向放在同一路由上对比，而不是脑内比较。

第三步，我把原型放进这个 Hexo 项目的新页面路径：

- `/music-player-prototype/?variant=A`
- `/music-player-prototype/?variant=B`
- `/music-player-prototype/?variant=C`

第四步，我让这个原型自包含，不依赖外部音频文件，而是直接用浏览器里的 Web Audio API 合成简单音色循环。这样它既能“像播放器一样动起来”，又不会引入额外素材和资源管理负担。

第五步，我额外写了一份原型记录页，说明这个原型回答了什么问题、用了哪些 skill、以及我实际验证到了什么。

### 这次操作更像哪些 skill

如果用 `mattpocock/skills` 的视角来看，这次动作大致对应下面几个 skill：

- `prototype`：这次最核心用到的就是它，而且是 UI 分支。它要求在一个路由里做多个结构差异明显的方案，并用切换器来回比对
- `zoom-out`：先看这个博客项目的结构，决定原型应该作为独立页面落在 `source/` 下，而不是硬塞进现有文章模板
- `handoff`：我把原型问题、变体设计、验证范围和限制，单独记录进 `NOTES` 页面，方便后续继续演进或删除
- `browser-use`：我专门检查了这个 skill，想用它做本地页面验证，但当前会话缺少它要求的执行入口，因此这次只能把它作为“已尝试接入但当前受限”的记录保留下来

这里最重要的一点是：我没有只“提到” skill，而是真的按 skill 指导的工作方式来做页面。

### 实际操作记录

做完这个音乐播放器原型后，一个更接近真实工程的问题其实是：

如果原型不符合预期，下一步该怎么处理？

我的处理方式不是继续在原型上盲改，而是先把“不符合预期”拆开。具体会按下面这套记录方式执行：

1. 先判断失败点是什么
   是视觉方向不对、布局层级不对、交互节奏不对，还是它其实回答错了问题。比如这个播放器原型，如果问题是“看起来不像博客组件”，那就不是去继续微调按钮，而是要重新定义原型目标。

2. 再判断哪些部分值得保留
   一个失败原型通常不会全废。可能 3 个 variant 都不对，但某个切歌区、某个状态展示方式、或者某种信息顺序是值得留下的。这些应该被单独记下来，而不是跟着整个原型一起被否掉。

3. 决定下一轮继续用哪个 skill
   如果问题还是“页面方向不对”，继续用 `prototype` 做下一轮变体。
   如果问题是“需求本身说不清”，回到 `grill-me`。
   如果问题不是样式，而是逻辑、状态或播放行为不对，应该切到逻辑验证或 `diagnose`，而不是继续做 UI 变体。

4. 不直接把失败原型继续演化成正式代码
   原型代码的目标是快，不是稳。即使某一版方向接近，也应该把结论吸收进正式实现，而不是继续在原型页上堆功能。

5. 最后把结论明确记录下来
   至少要记录三件事：
   这个原型原本在回答什么问题；
   它为什么不符合预期；
   下一轮准备验证什么。

拿这次音乐播放器举例，如果它不符合预期，我会优先追问这些问题：

- 3 个变体是都不对，还是其中一个方向对但细节不够好
- 目标更接近“独立播放器产品”，还是“博客里的展示型模块”
- 更在意视觉风格，还是更在意播放交互
- 下一轮是继续做轻量原型，还是开始往正式页面收敛

也就是说，真正的“实际操作记录”不只是记录我做了什么，而是记录：

原型失败之后，应该如何判断、如何保留结论、以及如何决定下一轮动作。

### 这次实践的反馈

这次实践比“写一段文档说明”更能说明 skill 的价值，因为它直接回答了一个设计问题：

这个项目里的轻量音乐播放器，至少可以先朝 3 个方向去看：

1. 封面驱动型
2. 控制台驱动型
3. 队列驱动型

而不是一上来就把某个单一页面写死。

从这次例子里，能看到 skill 化工作流最直接的几个好处：

1. 先看上下文，再决定改哪里，能减少瞎改配置
2. 先确认“要回答的问题是什么”，再决定做原型还是正式功能
3. 同一路由下做多个变体，比只做一个页面更容易比较方向
4. 把验证结果和限制写下来，后续接手的人不会误以为这个原型已经完成了完整浏览器验收

这次也暴露了一个很真实的工程事实：不是每个能用的 skill 都一定能在当前会话里完整跑通。比如这次我确实检查了 `browser-use`，但由于当前执行入口不可用，最终只能完成“构建验证 + 路由生成验证”，而不能假装已经做了完整的浏览器自动化验收。

这反而说明，skill 不是魔法按钮。真正有价值的是：

1. 知道应该用什么工作流
2. 能在可用条件下尽量贴近它执行
3. 把没做到的部分明确记录下来

如果后面要继续推进，这个原型最自然的下一步不是“继续堆代码”，而是先在 3 个变体里选一个主方向，再把它真正吸收成正式页面或组件。
